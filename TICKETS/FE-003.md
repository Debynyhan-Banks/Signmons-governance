# FE-003 - Reusable Marketing Components (Visual Parity)

## Ticket Summary

- Ticket ID: `FE-003`
- Repo: `Signmons-Marketing-Web-Feat-Marketing-Site`
- Screen IDs: `SCR-PUB-001`, `SCR-PUB-007`, `SCR-PUB-009`
- Objective: refactor the public marketing funnel to reusable React/MUI components while preserving current visual output and behavior.

## In Scope

- Refactor-only pass for:
  - `SCR-PUB-001` Home
  - `SCR-PUB-007` Demo Experience
  - `SCR-PUB-009` Contact / Waitlist / Demo Request
- Extract reusable presentation components for shared UI patterns:
  - page shell (nav/footer/background container)
  - section heading block (tag, heading, subheading)
  - CTA row/button variants
  - card/list primitives used repeatedly across the three screens
- Keep content in typed data contracts (`src/data/*`) and render via typed props.
- Preserve existing route behavior and CTA destinations for touched screens.
- Preserve current animation intent while moving animation hooks into reusable structure.

## Out of Scope

- Visual redesign (color, spacing, typography, motion language)
- New backend/API features or authentication changes
- New pages beyond listed screen IDs
- SEO/content strategy rewrite

## Technical Requirements

- React + TypeScript + MUI patterns only for touched surfaces.
- No inline style sprawl in page-level code; move styles into reusable component boundaries.
- No dead UI actions on touched screens.
- Keep component responsibilities narrow (presentation-only where possible).

## Acceptance Criteria (must all pass)

- [ ] Touched screens render with visual parity to baseline captures.
- [ ] Repeated UI structures are implemented as reusable components.
- [ ] Marketing copy for touched sections is sourced from typed data contracts.
- [ ] No route regressions or CTA target regressions on touched screens.
- [ ] Accessibility baseline preserved (heading order, keyboard access, focus states).
- [ ] Frontend gates executed with evidence:
  - `npm run -s build`
  - `npm run -s lint`
  - `npm test -- --runInBand` (or approved equivalent with explicit note)

## Evidence Required

- Baseline + post-refactor screenshots for each touched screen:
  - desktop full page
  - mobile full page
  - one close-up per major reusable component family
- Command output logs:
  - build, lint, test
- Short mapping note:
  - old template block -> new reusable component

## Risks

- Visual drift during extraction from static templates
- Hidden coupling between copied HTML and ad-hoc page scripts
- Gate failure if test command contract is missing in current branch

## Commit Convention

- Commit message: `refactor(marketing): implement FE-003 reusable components with visual parity`
- One focused commit only for this ticket
