# APP-012 - Payment Gate and Webhook Status Workflow

## Ticket Summary

- Ticket ID: `APP-012`
- Repo: `signmons-calldesk-backend` (+ frontend app surface)
- Screen IDs: `SCR-APP-006A`, `SCR-APP-006B`, `SCR-APP-006C`, `SCR-APP-023`, `SCR-CUST-002`
- Objective: operationalize payment-first booking with deterministic gate enforcement and webhook-status transparency.

## In Scope

- Payment link/deposit request management.
- Payment status and webhook event tracking:
  - paid
  - failed
  - expired
  - refunded/canceled
- Booking policy gate status:
  - booking unlocked/locked
  - required fee rules
- Customer payment/deposit status page.

## Out of Scope

- Storing card PAN data in Signmons systems.
- Processor-internal dispute lifecycle UI.
- Full accounting export workflows.

## Acceptance Criteria (must all pass)

- [ ] Payment links and deposit requests can be created and tracked.
- [ ] Webhook status is visible and fail-closed on validation errors.
- [ ] Booking gate enforces tenant policy before dispatch readiness.
- [ ] Customer payment status is viewable via secure link.
- [ ] Payment and gate actions are audit logged.

## Test and Evidence Plan

- Backend gates: `npm run -s build`, `npm test -- --runInBand`, `npm run -s arch:check`
- Frontend gates (if touched): `npm run -s build`, `npm run -s lint`, `npm test -- --runInBand`
- Evidence:
  - payment gate screenshots
  - webhook validation proof
  - booking lock/unlock proof

## Risks

- Webhook signature misconfiguration causing false states.
- Policy drift between UI and server enforcement.
- Customer confusion on partial/failed payment states.

## Commit Convention

- Commit message: `feat(app): implement APP-012 payment gate workflow`
- One focused commit only.
